📚 What you'll learn

Two different security checks are easy to confuse. This demo separates them so the difference becomes obvious.

Authentication Proving who you are. In this demo, the system correctly knows you are Bob — that part works fine.
Authorisation Deciding what that identity may do. This demo has no such check at all: being "you" is treated as enough to see anyone's record.

🚨 What just happened

🤔 Why did this happen?

The server almost certainly has code like getRegistration(id) that loads a record by its ID and returns it — full stop. It never asks the one question that matters: does the currently authenticated user actually have a legitimate reason to see this specific record? This pattern is sometimes called an insecure direct object reference (IDOR) or, more generally, broken access control — and it is consistently one of the most common real-world web vulnerabilities, precisely because authentication working correctly makes it easy to assume authorisation is also being handled.

✅ How to fix it

The server needs an explicit authorisation check on every request for a specific record — not just a check that someone is logged in:

// Vulnerable: loads and returns any record by ID, no ownership/role check function getRegistration(id) { return db.registrations.findById(id); } // Fixed: checks the caller is either the record's owner, or holds // a role authorised to view other members' records function getRegistration(id, currentUser) { const record = db.registrations.findById(id); if (record.userId !== currentUser.id && currentUser.role !== 'officer') { throw new AuthorisationError(403, 'Not permitted to view this record'); } return record; }

Notice the fix does not remove Bob's own access (record.userId !== currentUser.id is false for his own record) — it only blocks the specific case that should never have been allowed: one Member's session reading another user's data.

🎯 Key takeaway

"The user is logged in" and "the user may see this" are two different questions, and a system must answer both, on every request — not just once at login. A login screen with no per-request authorisation check is a locked front door with every window left open.
