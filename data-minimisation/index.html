Campus Club Membership Service

Event Check-in Dashboard (Data Minimisation Practical Demo)

📚 What you'll learn

This service's check-in page is designed to show only a student's name and attendance status. But a clean-looking interface is not proof that the server is only sending clean data. This demo shows the gap between what a page displays and what it actually receives — and why that gap is a real confidentiality failure, not a cosmetic one.

Data minimisation Collect and expose no more data than a stated purpose requires. Displaying only a name does not minimise anything if the full record still travels to the browser.
Confidentiality Information should be seen only by those with a legitimate need. Sending a full record to every visitor's browser breaks confidentiality even if the interface hides it.

🔍 Step 1 — Inspect it yourself (recommended)

Instructions: Open Developer Tools (F12 or right-click → Inspect), go to the Network tab, filter by Fetch/XHR, then click "Load Attendance List" below. Click the attendance.json request and open its Response tab to see the raw payload the server actually sent.

🔬 Step 2 — Reveal what the server actually sent

Already checked DevTools? Compare your findings against the annotated payload below — this is the exact response the server sends for every row shown above, using Alice's record as the example.

{ "displayName": "Alice Smith", "status": "Attended", "studentId": "66031024", "email": "alice.a@nu.ac.th", "allergies": "Peanuts (Severe anaphylaxis risk)", "disabilityAccommodation": "Needs sign language interpreter / front-row seating" }
studentId A unique identifier the check-in page never displays or uses — no reason for it to leave the server for this view.
email Contact information that could be harvested at scale simply by loading this page repeatedly.
allergies A medical detail. Exposing it to anyone who opens the Network tab is a serious, avoidable privacy harm.
disabilityAccommodation A sensitive accessibility need, disclosed to an audience (anyone loading the page) far wider than the "people arranging support" it was collected for.

🤔 Why did this happen?

Almost certainly, the backend queries "the whole student record" from its database because that is the easiest code to write, and simply serializes the entire row into the API response — trusting the front-end to only display the safe fields. But the front-end is not a security boundary: anyone can bypass it and read the raw response directly, exactly as you just did.

✅ How to fix it

The fix is not encryption, not a login wall, and not hiding the field harder in the UI — it is sending less data in the first place. Minimisation has to happen on the server, before the response is even sent:

❌ Vulnerable response

{ "displayName": "Alice Smith", "status": "Attended", "studentId": "66031024", "email": "alice.a@nu.ac.th", "allergies": "Peanuts...", "disabilityAccommodation": "..." }

✅ Minimised response

{ "displayName": "Alice Smith", "status": "Attended" }

If a different, authorised view genuinely needs the accessibility note (e.g. an event-support role preparing a room), that view should request it separately, through an endpoint that checks the caller's role — not receive it by default alongside every other field.

🎯 Key takeaway

"The interface only shows a name" is not a security control. If you cannot see who receives a response and confirm it contains only what that purpose needs, assume it is over-exposed. Minimisation is a decision made when the response is built, not a side-effect of a tidy-looking screen.
